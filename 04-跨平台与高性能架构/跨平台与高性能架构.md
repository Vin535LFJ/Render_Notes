## 1. 跨平台零拷贝物理链路 ✅已验证

### 1.1 零拷贝的本质

零拷贝（Zero-Copy）在多媒体渲染中的含义并非指像素数据在物理硬件上绝对静止，而是指**在数据流转链路上消除了 CPU 参与的物理内存复制操作**。其物理本质是借助操作系统内核的内存共享协议，让解码芯片（VPU）、着色器处理器（GPU）以及显示核心对同一块物理页面（Physical Page）进行统一编址与并发寻址 ✅已验证

然而，零拷贝并非绝对的物理保障。在以下场景中，硬件层内部仍可能静默发生 DMA 拷贝 ⚠️有条件成立：

- **格式错配**：当源数据格式（如 YUV 4:2:0 NV12）与目标采样器期望的格式不完全匹配时，GPU 采样器硬件（TMU）内部的寄存器仍需执行乘法累加计算来动态合成 RGBA 像素。这种在 GPU 内部、基于固定硬件功能执行的格式重构，在系统层面被视为零拷贝，但从硅片层面看依然伴随着电荷的转移与物理重构
- **跨硬件边界**：当数据需要跨越不同的硬件加速器边界（如从 VPU 到独立 GPU）时，若两者不共享同一物理内存控制器，硬件内部仍可能触发 DMA 引擎的静默拷贝
- **色域空间转换**：不同色彩空间矩阵（BT.601 / BT.709 / BT.2020）之间的转换，虽然由 GPU 固定硬件单元执行，但本质上仍是硬件级的信号重构操作

#### 架构抉择原则

1. **纯全屏视频播放业务**：应强硬执行 **Surface-direct（直出模式）**，将 Surface 绑定给解码器，使像素完全不经过 GPU 计算，由系统 `SurfaceFlinger` 的硬件 Overlay 物理层叠加渲染，此能效比最高
2. **特效播放器或带滤镜的业务**：必须妥协使用 **Texture-based（纹理化模式）**，绑定 `SurfaceTexture` 桥接为 OES 纹理，虽然增加了 GPU 采样开销，但获得了完备的可编程灵活性

### 1.2 Android 平台零拷贝

Android 平台的零拷贝链路基于 **MediaCodec + AHardwareBuffer + EGLImage** 机制实现 ✅已验证

#### 核心组件

| 组件 | 职责 | 物理特性 |
|------|------|----------|
| **MediaCodec** | 系统级硬解码 API，驱动 VPU 芯片执行解码 | 解码输出直接写入 Gralloc 缓冲区，不经过系统主存 |
| **AHardwareBuffer** | 由 Gralloc 分配的高性能显存缓冲区 | 支持跨进程共享与多 API 绑定，是零拷贝的物理载体 |
| **EGLImage** | 跨 API 的硬件纹理链接对象 | 将 AHardwareBuffer 映射为 GPU 可采样的纹理资源 |

#### 零拷贝绑定流程

**OpenGL ES 路径**：

1. 将 MediaCodec 解码的输出 Surface 关联到 `AHardwareBuffer`，解码完毕后该 Buffer 直接存放 VPU 输出的 YUV (NV12) 像素物理页
2. 通过 `eglGetNativeClientBufferANDROID` 将 `AHardwareBuffer` 转换为 `EGLClientBuffer`
3. 调用 `eglCreateImageKHR` 创建 `EGLImage` 硬件链接对象
4. 绑定为不透明的 OES 外部纹理：`glBindTexture(GL_TEXTURE_EXTERNAL_OES, texID)` → `glEGLImageTargetTexture2DOES(GL_TEXTURE_EXTERNAL_OES, eglImage)`
5. GPU 直接采样 BufferQueue 中的原生硬缓冲，跳过所有传统的数据回读（Readback）开销

```
EGLClientBuffer clientBuf = eglGetNativeClientBufferANDROID(ahbHandle);
EGLImageKHR eglImage = eglCreateImageKHR(eglDisplay, EGL_NO_CONTEXT,
    EGL_NATIVE_BUFFER_ANDROID, clientBuf, attrs);
glBindTexture(GL_TEXTURE_EXTERNAL_OES, texID);
glEGLImageTargetTexture2DOES(GL_TEXTURE_EXTERNAL_OES, eglImage);
```

**Vulkan 路径**：

通过 `VK_ANDROID_external_memory_android_hardware_buffer` 扩展，直接将 `AHardwareBuffer` 绑定为 `VkImage`，实现零拷贝图形渲染。该扩展允许 Vulkan 应用直接导入 Android 硬件缓冲区的外部内存，无需任何 CPU 侧的数据搬运

#### 物理数据流

```
音视频码流 → VPU 硬件解码器 → Gralloc 共享物理内存 (AHardwareBuffer)
                                    ↓
                    ┌───────────────┼───────────────┐
                    ↓               ↓               ↓
            OpenGL ES OES      Vulkan VkImage    SurfaceFlinger
            外部纹理采样       直接绑定采样      硬件 Overlay 合成
```

### 1.3 Web 平台零拷贝

Web 平台的零拷贝链路基于 **WebCodecs + WebGPU importExternalTexture** 机制实现 ✅已验证

#### 核心流程

1. **多线程异步硬解**：调用浏览器底层的 `WebCodecs`（利用系统 VPU 硬件硬解）导出 `VideoFrame` 对象。`VideoFrame` 直接关联 VRAM 物理页面，而非 JavaScript 堆内存
2. **外部纹理直接注入**：在 C++（编译至 WebAssembly）渲染循环中，严禁调用任何 `copyTo` 方法，直接将 JS 端的 `VideoFrame` 包装成不透明句柄传给 WebGPU 底层

```
WGPUExternalTextureDescriptor extTexDesc = {};
extTexDesc.source.videoFrame = jsVideoFrameHandle;
WGPUExternalTexture externalTexture =
    wgpuDeviceImportExternalTexture(device, &extTexDesc);
```

3. **着色器内并行色域转换**：在 WGSL 着色器中采样 `externalTexture` 时，YUV420 到 RGB 的色彩矩阵变换直接由 GPU 固定硬件单元内完成，转换耗时由 5ms（CPU 耗时）缩短至接近 0ms

#### 关键优势

- YUV→RGB 转换由 GPU 原生插值器零开销完成，无需片元着色器手动实现色彩矩阵运算
- 整体时延可压缩至 **2ms 以内** [测试数据-仅供参考]
- 完全消除了 JavaScript 堆 → WASM 线性内存 → GPU 驱动缓冲之间的多次 CPU 中转拷贝

#### 与传统 WebGL 路径对比

| 维度 | 传统 WebGL (`texImage2D`) | WebGPU 零拷贝 (`importExternalTexture`) |
|------|---------------------------|------------------------------------------|
| 数据路径 | VideoFrame → CPU 拷贝 → `texImage2D` → GPU 纹理 | VideoFrame → GPU 外部纹理直接映射 |
| CPU 参与 | 每帧需 CPU 执行像素搬运 | CPU 仅提交指令，不参与数据搬移 |
| 色域转换 | 需片元着色器手动实现 YUV→RGB | GPU 硬件原生插值器自动完成 |
| 单帧延迟 | 5~15ms | <2ms [测试数据-仅供参考] |

### 1.4 桌面端零拷贝

桌面端零拷贝因 GPU 厂商和操作系统不同，存在两条主要技术路径 ✅已验证

#### Linux: VAAPI 解码 → dmabuf → EGLImage

1. 使用 FFmpeg 初始化 VAAPI 硬解组件，设置 `AVCodecContext` 的 `hw_device_ctx` 指向 VAAPI 设备（`/dev/dri/renderD128`）
2. VAAPI 解码后输出一个指向 `dmabuf` 的物理文件描述符（File Descriptor）
3. 将 `dmabuf FD` 直接作为 `EGLImage` 绑定至 OpenGL 渲染管线
4. GPU 直接对 dmabuf 映射的物理显存进行纹理采样，无需 CPU 介入

```
[ VAAPI 解码 ] → dmabuf FD → EGLImage 绑定 → OpenGL 纹理采样 → 渲染输出
```

#### NVIDIA: NVDEC → CUDA 设备指针 → OpenGL/Vulkan 互操作

1. NVDEC 硬解后得到一个 CUDA 设备指针 `CUdeviceptr`，数据位于 GPU 显存
2. 注册 OpenGL 的 Texture 为 CUDA 资源：

```
cuGraphicsGLRegisterImage(&cudaResource, openGLTextureID,
    GL_TEXTURE_2D, CU_GRAPHICS_REGISTER_FLAGS_WRITE_DISCARD);
```

3. 在渲染帧循环中，通过 `cuGraphicsMapResources` 映射该资源，并利用 `cuMemcpy2D` 将解码后的 NV12 数据进行 **GPU 内部的物理拷贝（Device-to-Device，D2D）**
4. D2D 拷贝不经过系统内存总线，数据始终在显存内部流转，带宽极高且延迟极低

```
[ NVDEC 解码 ] → CUdeviceptr (显存) → cuMemcpy2D (D2D) → OpenGL 纹理 → 渲染
                                                  ↑
                                        不经过内存总线，纯显存内部拷贝
```

#### 桌面端零拷贝路径对比

| 维度 | VAAPI + dmabuf + EGLImage | NVDEC + CUDA Interop |
|------|---------------------------|----------------------|
| 适用平台 | Linux（Intel/AMD 集显） | Windows/Linux（NVIDIA 独显） |
| 数据拷贝类型 | 真正零拷贝（文件描述符共享） | D2D 显存内部拷贝（不经内存总线） |
| API 互操作 | dmabuf → EGLImage → OpenGL | CUDA → cuGraphicsGLRegisterImage → OpenGL |
| Vulkan 支持 | dmabuf → VkImage（通过外部内存扩展） | CUDA → Vulkan（通过外部内存/信号量扩展） |

## 2. RenderGraph 架构设计

### 2.1 核心价值

RenderGraph 通过将渲染管线中的各个 RenderPass 节点、物理纹理资源以及数据读写依赖关系抽象为一个**有向无环图（DAG, Directed Acyclic Graph）**，在框架层面接管了显存资源的生命周期 ✅已验证

#### 两大核心优化机制

**1. 物理显存复用（Aliasing）**

RenderGraph 在图编译阶段，通过遍历依赖路径，可以对生命周期互不相交的中间离屏纹理进行物理显存的最大化复用。具体机制：

- 分析 DAG 中每个纹理资源的首次写入（Write）和最后一次读取（Read）时间点，确定其活跃生命周期
- 生命周期不重叠的纹理可共享同一块物理显存区域
- 总显存开销从各 Pass 独立占用总和**陡降 40% 以上** [测试数据-仅供参考]

```
Pass A 输出纹理 T1 (生命周期: Frame 1-3)
Pass B 输出纹理 T2 (生命周期: Frame 4-6)
→ T1 与 T2 生命周期不相交 → 物理显存复用 (Aliasing)
→ VRAM 峰值从 (T1 + T2) 降至 max(T1, T2)
```

**2. 自动推导管线屏障（Pipeline Barriers）**

RenderGraph 在编译期自动推导 Vulkan/WebGPU 等 API 的最佳管线隔离屏障，避免了人工编写产生的显存冲突及死锁：

- 根据 DAG 的读写依赖关系，自动插入 `vkCmdPipelineBarrier` / `wgpuCommandEncoder.insertDebugMarker` 等同步原语
- 确保写入操作在读取之前完成（Write-Before-Read 依赖）
- 避免过度同步（不必要的屏障会降低 GPU 并行度）
- 消除人工配置屏障时极易引入的管线死锁与数据竞争

### 2.2 边界 ⚠️有条件成立

RenderGraph 并非在所有场景下都是最优选择，其适用性取决于渲染管线的复杂度：

**不宜使用的场景**：

- **单 Pass 简单渲染**：RenderGraph 增加 DAG 构建与图编译的 CPU 耗时，而单 Pass 场景下无显存复用红利，属于过度设计
- **Pass ≤ 2 且为单线结构**：直接手写双 FBO 环即可，引入 RenderGraph 只会带来图构建的冗余 CPU 计算负荷

**必须使用的场景**：

- **复杂树状混合分支**：多路画面剪辑、实时多层特效、带景深的后期处理管线等场景，必须选用 RenderGraph 架构
- **特效 Pass 深度 > 3**：涉及模糊（Blur）、描边等空间采样算法时，人工配置屏障极易出错，必须通过 RenderGraph 进行自动优化
- **多路并发渲染**：避免人工配置屏障错误引起的致命管线死锁

### 2.3 量化评估维度

客观量化 RenderGraph 收益时，不能单一依靠"画面帧率（FPS）"等非科学维度，必须建立四维量化监控体系 ✅已验证

| 评估维度 | 物理含义 | 测量工具 | 期望优化方向 |
|----------|----------|----------|--------------|
| **VRAM Peak**（显存占用峰值） | 渲染过程中物理显存的最大占用量 | RenderDoc 内存快照对比 | 下降 40%+ [测试数据-仅供参考] |
| **动态显存申请/释放频次** | 运行时 `vkAllocateMemory` / `glGenTextures` 的调用次数 | GPU Profiler / 驱动层 Hook | 趋近于零（Aliasing 复用） |
| **Vulkan/WebGPU 屏障注入时钟开销** | 管线屏障同步导致的 GPU 空闲气泡时间 | Nsight Graphics Timeline | 自动推导最优屏障，减少气泡 |
| **多帧平均 Overdraw** | 同一像素被重复写入的次数 | RenderDoc Overdraw 可视化 | 降低冗余 Pass 的像素填充 |

#### 评测实验设计原则

1. **硬件基准绝对统一**：标明具体的 SoC 芯片及底层显卡驱动版本
2. **输入测试用例标准化**：采用统一编码规格的标准码流作为唯一输入源
3. **内核节点物理监控**：使用 Systrace 抓取专属进程的 CPU 线程活跃周期，通过 ADB Shell 读取内核节点获取电流与温度变化，利用 `dumpsys meminfo` 过滤 `gpu_memory` 获取物理页显存占用

## 3. GPU 原生运行时（GPU-Native Runtime）

### 3.1 计算范式演进

GPU-Native Runtime 代表了计算架构的根本性范式转移 ✅已验证

#### 传统计算范式

```
编写 kernel → 分发（Dispatch） → GPU 执行 → 拷贝结果回 CPU → CPU 决策 → 再次分发...
```

- 数据在 CPU 与 GPU 之间反复搬运
- CPU 是控制中心，GPU 是计算加速器
- 每次 CPU-GPU 同步都会产生严重的总线延迟与控制依赖

#### GPU-Native 计算范式

```
数据常驻 GPU → GPU 自主执行 → GPU 自主决策 → 结果原地更新 → 仅在 I/O 边界与 CPU 交互
```

- **数据常驻 GPU 为默认行为**，CPU 退化为 I/O 控制器
- GPU 不仅负责计算，还承担调度、决策与控制逻辑
- 消除了 CPU-GPU 之间的运行期控制依赖与同步时延

#### 范式对比

| 维度 | 传统范式 | GPU-Native 范式 |
|------|----------|-----------------|
| 数据驻留 | CPU 主存为默认位置，按需拷贝至 GPU | GPU 显存为默认位置，仅在 I/O 时与 CPU 交互 |
| 控制流 | CPU 主控，GPU 从属 | GPU 自主执行控制逻辑 |
| 同步开销 | 每次 kernel 启动都有 CPU-GPU 同步 | 批量提交，运行期内无同步 |
| 适用场景 | 通用计算、简单任务 | 高密度并行计算、LLM 推理、物理仿真 |

### 3.2 GPU 原生编译架构 [AI-生成]

GPU 原生编译技术使用多层编译管道，直接将高阶域语言（DSL）编译为硬件可直接执行的代码，是 GPU-Native Runtime 的核心基础设施。

| 编译框架 | 输入语言 | 编译管道 | 产出代码 |
|----------|----------|----------|----------|
| **MC/DC**（蒙特卡洛计算） | Python | Numba (JIT 编译) → Harmonize → NVCC/HIPCC 工具链 | NVIDIA PTX 汇编 / AMD GCN/HSA 内核 |
| **DPIA**（安全数据并行） | 函数式数据语言 | 连续传递变换 → 静态数据防竞争重写 → 降级为命令式 IR | OpenCL / CUDA 原生 C 级内核代码 |
| **GraphIt/GG**（图论计算） | 算法 DSL + 调度 DSL | SSA IR 静态单赋值图层分析 → 自动依赖消减与原子操作注入 | 经过融合（Fused）的高度优化 CUDA 内核 |
| **xQC**（量子化学 JIT 计算） | Python / 模板 C++ | NVRTC（运行期运行时代码生成器） | 针对特定原子轨道、阶数高度特异化的 PTX 缓存高速内核 |

#### 编译架构共性特征

1. **多层 IR 管道**：高级 DSL → 中间表示（IR）→ 硬件原生代码，每层执行特定优化
2. **领域特定优化**：针对特定计算领域（蒙特卡洛、图论、量子化学）的算法级优化
3. **运行时代码生成**：JIT 编译允许根据运行时参数动态生成最优代码
4. **自动并行化**：编译器自动推导数据并行性，屏蔽显式线程管理

### 3.3 前沿架构突破 [AI-生成]

#### Gram2Token 语法约束解码

传统的约束解码需要将 GPU 推理得到的 Token 读回到 CPU，执行复杂的上下文文法约束解析生成掩码，再拷回 GPU 引导下一次采样，这造成了极高的控制依赖和同步时延。

Gram2Token 的核心突破：

- 将约束状态预编译为 Token 级的表格映射（O(1) Table Lookups）
- 直接在 GPU 侧以 Native 方式执行约束条件检查
- 彻底斩断 CPU 与 GPU 之间的运行期依赖
- 使 LLM 结构化输出（如 JSON 格式对齐）的吞吐率成倍暴涨

```
传统路径：GPU 生成 Token → CPU 读回 → CPU 约束检查 → 掩码回传 GPU → 下一次采样
Gram2Token：GPU 生成 Token → GPU 侧 O(1) 约束查表 → GPU 侧直接掩码 → 下一次采样
```

#### 物理仿真 GPU 原生运行时

多相流等高密集度物理动力学仿真，在完全部署于 GPU 并行核心计算后：

- **单 GPU 等效 160~512 个 CPU 核心**的计算通量（CPU Core Equivalence）
- 在确保计算结果物理一致的前提下，GPU 原生计算所消耗的总能量仅为 CPU 集群等效运行能耗的 **19%~35%**
- 展现出极佳的能效比，适用于 CFD（计算流体力学）、多相流模拟等高密度科学计算场景

## 4. AI 推理与图形渲染零拷贝互操作

### 4.1 ncnn/Vulkan 物理共享上下文

ncnn 是腾讯高性能端侧 AI 框架，深度支持 Vulkan 后端。通过物理共享 Vulkan 上下文，可实现渲染引擎与 AI 推理引擎之间的零拷贝互操作 ✅已验证

#### 共享机制

1. **共享上下文**：在初始化时，让渲染引擎的 Vulkan 上下文与 ncnn 共享相同的 `VkInstance`、`VkPhysicalDevice` 和 `VkDevice`
2. **外部内存直接绑定**：当 Compute Shader 完成视频画面渲染后，将对应的 `VkDeviceMemory`（图形纹理显存）句柄直接导出
3. **无拷贝推理**：在 ncnn 侧直接使用该图形显存句柄来初始化 `ncnn::Mat` 输入 Tensor，AI 推理内核直接在物理同源的显存区域内运行
4. **推理结果原地更新**：推理结果（如抠像生成的 Alpha 遮罩）在显存内原地更新，直接作为下一阶段图形 Shader 的采样输入

```
┌─────────────────────────────────────────────────────────┐
│              共享 Vulkan 上下文                           │
│  VkInstance / VkPhysicalDevice / VkDevice               │
│                                                         │
│  ┌──────────────┐    VkImage 句柄    ┌──────────────┐  │
│  │  渲染管线     │ ──────────────→  │  ncnn 推理    │  │
│  │  (FBO 输出)  │                   │  (ncnn::Mat)  │  │
│  └──────────────┘                   └──────┬───────┘  │
│         ↑                                   │          │
│         │         Alpha 遮罩 (原地更新)      │          │
│         └───────────────────────────────────┘          │
└─────────────────────────────────────────────────────────┘
```

#### 关键技术要点

- 渲染管线 FBO 最终生成的 `VkImage` 物理句柄，可以直接作为输入绑定给 `ncnn::Mat`
- 推理结果无需经过内存读回（Readback），在显存内原地更新
- 整个过程数据始终在 GPU 显存内部流转，CPU 零参与

### 4.2 LiteRT/AHardwareBuffer 桥接

Android 平台通过 LiteRT（TensorFlow Lite 继任者）与 AHardwareBuffer 的桥接机制，实现 AI 推理与图形渲染的零拷贝互操作 ✅已验证

#### 桥接流程

1. **硬件张量分配**：在初始化 AI 模型时，调用 `ANeuralNetworksMemory_createFromAHardwareBuffer` 直接把分配的 `AHardwareBuffer` 注册为 LiteRT 模型的输入/输出内存区域
2. **零转录管道**：
   - **Step 1**：多媒体子系统硬解视频，直接把 YUV/RGBA 数据写入此 `AHardwareBuffer`
   - **Step 2**：GPU 的 Compute Shader 对其进行特效前处理，并直接就地写入 `AHardwareBuffer`
   - **Step 3**：调用 `liteRT.run()` 执行硬件 GPU 委托推理，无需任何 CPU-GPU 之间或不同 GPU 上下文之间的数据转录

```
[ VPU 硬解 ] → AHardwareBuffer ← [ GPU Compute Shader 特效前处理 ]
                    ↓
         [ LiteRT GPU Delegate 推理 ] (零拷贝读取同一块物理内存)
                    ↓
         [ 推理结果原地写入 AHardwareBuffer ]
                    ↓
         [ GPU 渲染管线采样输出 ]
```

#### 性能收益

- 消除了主存与显存的读回开销
- 推理吞吐量实现数倍增长
- 1080P 实时智能抠像、背景替换特效的整体渲染管线可流畅稳定运行在 60 FPS 满帧 [测试数据-仅供参考]

### 4.3 全链路闭环

通过上述零拷贝互操作机制，最终实现从解码到呈现的全 GPU 闭环 ✅已验证

```
GPU 解码 → GPU 特效 → GPU 推理 → GPU 呈现
  (VPU)    (Compute     (ncnn/     (Framebuffer
            Shader)     LiteRT)    Output)
```

#### 全链路零拷贝保障

| 阶段 | 数据位置 | 拷贝开销 | 关键技术 |
|------|----------|----------|----------|
| GPU 解码 | VPU → Gralloc/AHardwareBuffer | 零拷贝 | MediaCodec / WebCodecs / NVDEC |
| GPU 特效 | GPU 显存内部流转 | 零拷贝 | OES 纹理 / importExternalTexture |
| GPU 推理 | 共享 Vulkan 上下文 / AHardwareBuffer | 零拷贝 | ncnn Vulkan 互操作 / LiteRT GPU Delegate |
| GPU 呈现 | Framebuffer → 屏幕 | 零拷贝 | SurfaceFlinger Hardware Overlay |

#### 传统方案对比

传统框架在 AI 特效介入时，会将像素回读到 CPU 再喂给模型，这会产生高达 15ms 的延迟并挤爆主存通道。全链路零拷贝闭环彻底消除了这一瓶颈：

- 传统路径：GPU 渲染 → `glReadPixels` 回读 CPU → CPU 喂入 AI 模型 → AI 输出回传 GPU → 渲染（延迟 15ms+）
- 零拷贝路径：GPU 渲染 → 显存内直接绑定 AI 输入 → AI 推理原地更新 → GPU 直接采样（延迟 <1ms）

## 5. WebGPU vs WebGL 架构演进

### 5.1 核心差异对比

WebGPU 非 WebGL 小改版，而是更接近现代显式 GPU API（Vulkan/Metal/D3D12）的新模型 ✅已验证

#### 架构级差异

| 关键技术维度 | WebGL (OpenGL ES 标准) | WebGPU (Vulkan/Metal 标准) |
|-------------|----------------------|---------------------------|
| **API 状态模型** | 全局、可变状态机模式。修改混合因子或深度测试需调用多个独立 API，极易产生状态污染 | 管道状态对象（PSO），无状态设计。所有状态在创建时不可变地封装在管线对象中，切换状态只需切换管线对象 |
| **执行模型** | 阻塞、同步渲染指令队列。`gl.getError` / `gl.readPixels` 需阻塞 CPU 等待 GPU 进程返回 | 纯异步执行与后台错误上报。所有资源创建和数据交互基于 Promise 异步模型，确保命令提交管道畅通 |
| **通用计算** | 缺乏原生支持，需通过纹理写入与变换反馈模拟 GPGPU | 原生支持一等公民——计算着色器（Compute Shader） |
| **着色器语言** | GLSL（类 C 语法，弱类型） | WGSL（类 Rust 语法，强类型，编译期安全检查） |
| **Z 裁剪轴范围** | [-1, 1] | [0, 1]（与物理屏幕深度对齐） |
| **渲染目标 Y 轴方向** | 向上（Upwards） | 向下（Downwards，与物理屏幕坐标系对齐） |
| **驱动优化时机** | 运行时状态验证，每次 DrawCall 都需驱动校验 | 管线创建时离线编译与优化，运行时零验证开销 |

#### 深层架构解析

**1. 全局状态机 vs PSO**

WebGL 本质上是一个巨大的全局状态机，其设置全局生效且极易产生状态污染。修改一个混合因子或深度测试开关，需要调用多个独立的状态配置 API。WebGPU 彻底移除了全局状态，采用无状态（Stateless）设计。它引入了渲染管道（RenderPipeline）和计算管道（ComputePipeline）的概念，将着色器、混合、深度测试、多重采样等所有状态在创建时直接不可变地（Immutable）封装在管道对象中。若要改变渲染状态，只需切换当前的管线对象。这使得驱动程序能够在管线创建时对其进行彻底编译和离线优化，完全规避了运行时的状态验证开销。

**2. 阻塞式 vs 异步式**

WebGL 设计中存在大量同步调用接口。获取当前渲染错误（`gl.getError`）或读取像素数据（`gl.readPixels`）需要阻塞 CPU 进程，等待跨进程通信（IPC）将指令传递给 GPU 进程，再等 GPU 硬件执行完毕并返回结果，导致 CPU 产生严重的等待空转。WebGPU 是一套完全异步的 API，其所有资源创建和数据交互（如 `mapAsync` 读取显存、异步着色器编译等）都基于 Promise 异步模型设计，即使发生创建错误也会在后台以异步静默方式上报，确保跨进程的命令提交管道畅通无阻。

### 5.2 WebGPU Compute Shader

WebGPU 计算着色器允许开发者将通用的、非图形学的并行计算任务直接分发至 GPU 执行，是 WebGPU 相对于 WebGL 最重大的能力升级之一 ✅已验证

#### 计算工作组与调用机制

当调用 `dispatchWorkgroups(X, Y, Z)` 时，GPU 将创建一三维工作组网格。而在 WGSL 内部，每个工作组内部又定义了局部的线程布局 `@workgroup_size(x, y, z)`。

全局调用索引的计算公式：

$$\text{global\_invocation\_id} = \text{workgroup\_id} \times \text{workgroup\_size} + \text{local\_invocation\_id}$$

| 内置标识符 | 含义 |
|-----------|------|
| `workgroup_id` | 当前工作组在全局网格中的三维坐标 |
| `local_invocation_id` | 当前线程在所在工作组内部的三维局部坐标 |
| `global_invocation_id` | 当前线程在全局计算任务中的唯一三维绝对坐标 |
| `local_invocation_index` | 将局部三维坐标线性化展平后的一维线程索引 |

#### 视频帧处理 WGSL 示例

以下 WGSL 计算着色器实现了视频帧的亮度调整与对比度滤波，直接对 GPU 存储缓冲区中的 Y 原始像素阵列进行高度并发的逐点操作：

```wgsl
@group(0) @binding(0) var<storage, read_write> inputY: array<f32>;
@group(0) @binding(1) var<storage, read_write> outputY: array<f32>;

struct FilterParams {
    brightness: f32,
    contrast: f32,
};
@group(0) @binding(2) var<uniform> params: FilterParams;

@compute @workgroup_size(256, 1, 1)
fn filterBrightness(
    @builtin(global_invocation_id) global_id: vec3<u32>
) {
    let index = global_id.x;

    if (index >= arrayLength(&inputY)) {
        return;
    }

    let raw_y = inputY[index];

    var processed_y = (raw_y - 0.5) * params.contrast + 0.5 + params.brightness;

    outputY[index] = clamp(processed_y, 0.0, 1.0);
}
```

#### 与 OpenGL Compute Shader 的差异

| 维度 | OpenGL Compute Shader | WebGPU Compute Shader |
|------|----------------------|----------------------|
| **着色器语言** | GLSL（`#version 430`+） | WGSL（强类型，编译期安全） |
| **工作组声明** | `layout(local_size_x=256) in;` | `@workgroup_size(256, 1, 1)` |
| **资源绑定** | `layout(binding=0)` + 全局状态机 | `@group(0) @binding(0)` + 显式绑定布局 |
| **存储缓冲区** | Shader Storage Buffer Object (SSBO) | `var<storage, read_write>` |
| **同步机制** | `barrier()` / `memoryBarrier()` | `workgroupBarrier()` / `storageBarrier()` |
| **原子操作** | `atomicAdd` 等函数 | `atomicAdd` 等函数（类型安全） |
| **调度方式** | `glDispatchCompute(x, y, z)` | `passEncoder.dispatchWorkgroups(x, y, z)` |
| **错误处理** | 同步 `glGetError`，易阻塞管线 | 异步验证，不阻塞命令提交 |

#### 内存竞争与同步

由于 GPU 的高度并发性，成千上万个线程在没有同步机制的情况下同时读写同一块显存（Storage Buffer），将产生灾难性的内存竞争与写覆盖。WebGPU 提供的同步机制：

- `workgroupBarrier()`：强制工作组内所有线程同步对齐，确保之前的写入对所有线程可见
- `storageBarrier()`：确保存储缓冲区的写入对后续操作可见
- 原子操作类型（如 `atomic<u32>`）：对共享变量进行原子增减操作，避免数据竞争
