## 图像渲染与引擎优化全维度学习指南

### 第一阶段：底层基石（数据流转全路径）

理解像素是如何从磁盘“长征”到屏幕的。

- **资源准备期（IO 与解码）：**
  - **磁盘 IO**：PNG/WebP 的文件读取。WebP 优化的是磁盘空间，而非运行内存。
  - **内存解码**：数据进入 Native Heap。注意：1080P 分辨率的图片无论压缩率多高，解码后在内存中固定占用约 **7.91MB** ($1920 \times 1080 \times 4$)。
- **逻辑提交期（CPU 决策）：**
  - **VBO/EBO 预上传**：提前将几何坐标（VBO）和索引（EBO）存入显存，避免每帧重复传输。
  - **DrawCall 产生**：CPU 向 GPU 下达的一次绘制指令。
- **显卡渲染期（GPU 自动化）：**
  - **顶点着色器**：坐标变换 ($MVP$ 矩阵计算)。
  - **光栅化**：矢量图形转为像素点阵（片元）。
  - **片元着色器**：执行**纹理采样**。这是下采样（1080P $\rightarrow$ 720P）发生的真正位置。
  - **混合与测试**：处理透明度（Alpha Blending）和前后遮挡（Depth Test）。

------

### 第二阶段：性能优化的“三剑客”

针对 CPU、GPU 和带宽的不同瓶颈进行专项突破。

| **优化手段**        | **作用阶段**  | **核心原理**               | **解决的问题**                                      |
| ------------------- | ------------- | -------------------------- | --------------------------------------------------- |
| **合图 (Atlas)**    | 准备期/提交期 | 减少 `glBindTexture` 调用  | **状态切换 (State Change)** 导致的 GPU 流水线停顿。 |
| **合批 (Batching)** | 逻辑提交期    | 合并顶点数据，减少指令发送 | **CPU 过载**（DrawCall 次数过多）。                 |
| **纹理压缩 (ASTC)** | 传输期/采样期 | 显存内直接存储压缩格式     | **显存带宽不足** 与 **显存容量溢出**。              |

------

### 第三阶段：视频与帧动画的深度差异

这是你当前项目的核心业务场景。

- **视频渲染**：利用 **VDEC（硬件解码器）** 走专用 Overlay 路径，通常 1 个物体（全屏矩形），1 次 DrawCall。
- **帧动画（Sprite）**：依赖 CPU/GPU 配合。
  - **无优化**：每帧切换纹理，每帧一次 DrawCall。
  - **优化后（合图+合批）**：多帧动作在一次 DrawCall 内完成。
- **全屏场景动画 (`secene_actions`)**：
  - **本质**：它是“伪动画”，在渲染看来就是一个全屏矩形（Quad）。
  - **瓶颈**：不在 DrawCall，而在**解码速度**与**纹理上传带宽**。

------

### 第四阶段：低端机顶盒（2G 内存）实战策略

针对你当前 720P 屏幕项目的避坑指南。

1. **分辨率红线**：
   - **死守 720P**。拒绝 1080P，因为 1080P 会多消耗 **2.25 倍** 的显存和解码带宽，且在 720P 屏幕上产生不必要的下采样计算。
2. **合图尺寸安全区**：
   - **限制在 2048x2048 以内**。推荐使用 **1x2 垂直合图 ($1280 \times 1440$)**，这是兼容性与 IO 效率的最佳平衡点。
3. **内存管理**：
   - **覆盖逻辑优化**：播放全屏场景动画时，立即销毁下层 `backgroundImageView`。
   - **异步解码**：使用独立线程池处理 WebP 解码，防止阻塞 GL 渲染线程（UI 掉帧）。
4. **纹理复用**：
   - 使用 **Ping-Pong Buffer**（双缓冲纹理）。不要频繁销毁和创建纹理 ID，而是使用 `glTexSubImage2D` 循环更新内容。

### 第五阶段：程序员的技能树（Shader 与 工具）

- **Shader 编程**：主要是编写 **GLSL**。重点掌握片元着色器，它是实现“滤镜、转场、特效”的魔法棒。
- **离屏渲染 (FBO)**：掌握“先画到纹理，再处理纹理”的技术，这是实现复杂后处理特效（如高斯模糊）的必经之路。
- **性能监控工具**：
  - **Android Profiler**：观察内存抖动。
  - **GPU Inspector / RenderDoc**：查看真实的 DrawCall 次数和状态切换开销。

------

### 💡 核心箴言

- **合图** 是为了让 GPU “换装”更少。
- **合批** 是为了让 CPU “喊话”更少。
- **720P** 是低端硬件的生存底线。
- **DrawCall** 只是表象，**状态切换** 和 **内存带宽** 才是深层瓶颈。





### 1. 为什么会有 DrawCall 这个概念？

在电脑或手机里，CPU 和 GPU 是两个独立的处理器。CPU 负责计算逻辑（猫往哪走），GPU 负责画图。它们之间通过一条“大路”（总线）连接。

CPU 不能直接把画面塞给 GPU，它必须通过调用图形接口（如 OpenGL 的 `glDrawElements` 或 Direct3D 的 `DrawIndexedPrimitive`）来发送指令。**每调用一次这种函数，就产生了一个 DrawCall。**

------

### 2. DrawCall 的“包裹”里装了什么？

一个 DrawCall 指令发出去时，CPU 已经提前在显存里摆好了三样东西，DrawCall 就像是启动开关：

- **模型（原材料）**：VBO/EBO 里的坐标点（告诉 GPU 形状）。
- **材质/纹理（油漆）**：Texture（告诉 GPU 颜色）。
- **Shader（施工说明书）**：着色器代码（告诉 GPU 怎么涂色、怎么处理光照）。

------

### 3. 为什么 DrawCall 多了会卡？（重点）

很多人误以为 DrawCall 多了是 GPU 累，**其实累的是 CPU**。

- **准备开销**：每次发送 DrawCall 前，CPU 都要进行大量的准备工作：检查状态、转换坐标、把数据打包、通过总线发送。
- **通信开销**：这就好比你让装修工刷墙，如果你说：“刷这块砖！”（DrawCall 1），“刷隔壁那块砖！”（DrawCall 2）……装修工（GPU）其实不累，但项目经理（CPU）喊话会喊到嗓子哑掉，而且每次喊话之间会有大量的等待和通信延迟。

**所以，优化的核心就是：** 尽量让 CPU 喊一句：“把这一整面墙都刷了！”（合并后的 DrawCall），而不是分一百次喊。
