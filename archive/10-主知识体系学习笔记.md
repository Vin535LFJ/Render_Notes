# 音视频 + 图形渲染 + 引擎 主知识体系学习笔记（按10章）

> 说明：本文基于仓库已有 6 份核心原始笔记融合整理，目标是用于面试学习的“具体知识文档”，不是目录或台账。

## 第1章 音视频基础

### 1.1 媒体处理三层数据

1. **压缩层**：H.264/H.265/AAC 等码流，体积小、不可直接显示。  
2. **原始层**：视频常见 YUV，音频常见 PCM。  
3. **显示层**：GPU 纹理/屏幕像素（通常 RGB）。

核心理解：播放器并不是“解码完就结束”，解码后仍需进入渲染链路。

### 1.2 YUV、RGB、GPU 纹理的关系

- YUV 是视频压缩与解码常见的中间表示。  
- 屏幕显示与多数图形管线最终按 RGB 输出。  
- GPU 纹理是图形 API 的资源对象，承载“可采样数据”，并非等价于某个固定像素格式。

### 1.3 时间戳与同步基础

- **PTS**：帧应该被显示/播放的时间。  
- **DTS**：帧应该被解码的时间顺序。  
- 有 B 帧重排序时，DTS 与 PTS 可能不一致。  
- 音画同步常以音频时钟为主，但并非唯一策略（也可能采用外部主时钟）。

### 本章面试重点

1. 为什么 B 帧会导致 DTS != PTS？  
2. 为什么“音画同步”本质是时钟问题而不只是 PTS 比较？

## 第2章 编解码与容器

### 2.1 容器、码流、解封装、解码

- 容器（MP4/MKV/FLV）负责组织轨道和时间信息。  
- 码流（H.264/H.265/AAC）负责压缩语法。  
- 标准链路：**Demux -> Decode -> Render**。

### 2.2 MediaCodec 与 FFmpeg 的职责边界

- **MediaCodec**：Android 编/解码 API，本身不负责解封装。  
- **FFmpeg**：框架集合，`libavformat/libavcodec/libavfilter/libswscale` 等分工明确。

### 2.3 硬解 vs 软解（正确表述）

- 硬解通常在移动端更节能、CPU 占用更低。  
- 软解通常兼容性更强、调试可控性更高。  
- “硬解一定更快”“软解一定更慢”属于过度绝对化，需看设备、码率、分辨率、驱动和实现路径。

### 2.4 推荐工程组合

常见工程策略：**FFmpeg 负责 demux，MediaCodec 优先硬解，失败回退软解**。  
这不是唯一“标准答案”，而是实务中常见折中。

### 本章面试重点

1. 你如何设计硬解失败回退软解的状态机？  
2. 为什么 FFmpeg 不能简单等同于“纯软解”？

## 第3章 OpenGL / OpenGL ES

### 3.1 OpenGL ES 的工程本质

- OpenGL ES 是状态机 API。  
- CPU 负责组织资源和提交 DrawCall。  
- GPU 并行执行着色和光栅流程。

### 3.2 Shader 的作用

- 顶点着色器：坐标/几何变换。  
- 片元着色器：像素级计算（YUV->RGB、滤镜、遮罩、颜色变换）。

### 3.3 DrawCall 的正确理解

- DrawCall 多通常会增加 CPU 提交负担。  
- 但瓶颈不只 DrawCall：状态切换、带宽、过度绘制、片元复杂度同样关键。

### 本章面试重点

1. 为什么“DrawCall 降低了但帧率不变”可能发生？  
2. 片元着色器复杂度如何影响移动端功耗？

## 第4章 渲染管线

### 4.1 管线阶段（简化）

顶点输入 -> 顶点处理 -> 图元装配 -> 光栅化 -> 片元处理 -> 测试/混合 -> 输出。

### 4.2 离屏渲染 FBO

- FBO 让渲染结果先进入纹理/缓冲，而非直接上屏。  
- 适合后处理、特效链、合成流程。  
- 多 Pass 常用 ping-pong 双缓冲规避读写冲突。

### 4.3 过度绘制（Overdraw）

- 同一像素被多次覆盖会增加片元阶段和带宽压力。  
- 对透明叠加和 2D 动画尤为敏感。

### 本章面试重点

1. FBO 为什么是视频特效链路的基础设施？  
2. 如何判断瓶颈在顶点阶段还是片元阶段？

## 第5章 Android 渲染系统

### 5.1 Surface 体系

- **Surface**：底层缓冲目标。  
- **SurfaceView**：独立 Surface 路径，偏低延迟展示。  
- **TextureView**：在 View 层级中合成，支持变换/叠加，灵活但通常更重。  
- **SurfaceTexture**：连接生产者缓冲与 OES 外部纹理采样。

### 5.2 直出与纹理化

- 直出：链路短，通常低开销。  
- 纹理化：可编程强，可做滤镜/美颜/合成。  
- 任何“固定延迟区间”“全程零拷贝”都必须加设备与系统版本条件。

### 本章面试重点

1. TextureView 为什么灵活但可能更耗合成成本？  
2. SurfaceTexture 在播放器中的关键价值是什么？

## 第6章 图像处理

### 6.1 YUV->RGB 转换

- 常见做法是在 GPU shader 中做转换，减少 CPU 压力。  
- 需要注意色彩空间（BT.601/709/2020）和 full/limited range。

### 6.2 Alpha 视频

- 常见工程方案：双通道或拼接通道，在 shader 合成 RGBA。  
- 跨平台 codec 对 alpha 支持不一致，需做能力矩阵。

### 6.3 典型图像处理链

解码输出 -> 纹理采样 -> 颜色转换 -> 特效处理 -> 混合输出。

### 本章面试重点

1. 为什么同一段视频在不同手机颜色观感会偏差？  
2. Alpha 合成常见边缘黑边问题怎么处理？

## 第7章 性能优化

### 7.1 性能问题的三层定位

1. CPU 提交层：DrawCall、状态切换、命令组织。  
2. GPU 渲染层：片元复杂度、overdraw、纹理采样。  
3. 内存/总线层：上传、拷贝、缓存失配、带宽瓶颈。

### 7.2 常见优化手段

- 合批：减少 CPU 提交次数。  
- 合图：减少纹理绑定切换（但需控制图集碎片）。  
- 压缩纹理：降低显存与带宽压力。  
- 控制 FBO 层数与分辨率：避免离屏开销过高。

### 7.3 定量结论的正确态度

任何“CPU 低多少/GPU 低多少/延迟是多少”的说法，都应绑定：
- 机型与系统版本
- 分辨率、码率、帧率
- 场景负载（特效数量）
- 采样窗口与统计口径

### 本章面试重点

1. 你如何区分是“GPU 忙”还是“CPU 忙”？  
2. 合图什么时候可能让性能变差？

## 第8章 播放器架构

### 8.1 统一播放主链路

数据源 -> demux -> 解码选择 -> 渲染输出 -> 同步校正。

### 8.2 关键架构能力

- 硬解优先 + 软解兜底  
- seek 后快速首帧恢复  
- 回退状态机避免黑屏/爆音  
- 队列背压避免延迟积累

### 8.3 多路播放

- 多路系统需统一时钟与队列策略。  
- 只对齐 PTS 往往不够，还要考虑时钟漂移与负载抖动。

### 本章面试重点

1. 软硬解切换如何做到用户几乎无感？  
2. 多路同步为什么不能只看“同一时刻的 PTS”？

## 第9章 引擎架构

### 9.1 RenderGraph 的价值与边界

- 价值：显式描述 Pass 依赖、资源生命周期、同步关系。  
- 边界：不是引入后必然提速，小项目可能增加管理开销。

### 9.2 跨平台“零拷贝”

- 本质依赖平台共享缓冲机制（Android AHardwareBuffer、iOS IOSurface/CVPixelBuffer 等）。  
- 目标是最小化拷贝，不是脱离平台语义的“绝对 0-copy”。

### 9.3 Web 路径

- WebCodecs + WebGPU 能力受浏览器版本与实现差异影响，需版本矩阵验证。

### 本章面试重点

1. RenderGraph 主要解决性能问题还是复杂度问题？  
2. 跨平台零拷贝的“成功标准”应如何定义？

## 第10章 项目实践（面试表达）

### 10.1 项目复盘模板

1. 目标：低延迟/低功耗/高兼容。  
2. 旧问题：CPU 高、拷贝多、渲染阻塞。  
3. 方案：解码链路改造 + 渲染链路重构 + 回退策略。  
4. 验证：统一压测口径（设备矩阵 + 指标 + 样本量）。  
5. 结果：量化收益 + 边界条件 + 未解决问题。

1. 为什么说“零拷贝”是目标不是保证？

3. SurfaceView 与 TextureView 的系统层差异是什么？  
4. 为什么 DrawCall 少了但不一定更快？  
5. YUV->RGB 在 CPU 与 GPU 上做的权衡是什么？  
6. 如何设计软硬解回退不闪屏？  
7. FBO 多 Pass 链路怎样避免过高内存与带宽开销？  
8. 多路同步遇到时钟漂移如何处理？  
9. RenderGraph 的收益如何量化？  
10. 如何构建“可复现实验”来证明优化有效？

## 附：统一结论口径（避免幻觉/绝对化）

- “一定更快/一定更省电/一定低延迟” -> 改为“通常/在某类场景下/需实测验证”。  
- “全程零拷贝” -> 改为“以减少CPU侧拷贝为目标，是否0-copy依赖平台实现”。

这套口径在面试中会显著提升可信度与工程成熟度。
