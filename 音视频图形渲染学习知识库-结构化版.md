## 3. 高频知识点验真与纠偏（第一轮）

## 3.1 解码/封装/渲染职责
- 结论：`MediaCodec 只负责编解码，不负责解封装`。`✅已验证`
- 结论：`FFmpeg 可解封装+软解+（可选）硬件加速接口`。`✅已验证`
- 易错点：把“播放器”与“解码器”混为一谈。

## 3.2 SurfaceView vs TextureView
- 结论：`SurfaceView` 通常更省功耗、低延迟（独立 Surface 合成路径）。`⚠️有条件成立`（依赖设备合成策略）
- 结论：`TextureView` 更易做 UI 变换，但常有额外合成成本。`⚠️有条件成立`
- 易错点：把结论绝对化为“TextureView 一定慢很多”。

## 3.3 OES 外部纹理与零拷贝
- 结论：`SurfaceTexture + GL_TEXTURE_EXTERNAL_OES` 常用于视频帧直达 GPU 采样。`✅已验证`
- 易错点：宣称“全链路必然零拷贝”。实际仍可能出现驱动/格式转换拷贝。`⚠️有条件成立`

## 3.4 时间戳与同步
- 结论：音视频同步主要围绕主时钟（通常音频时钟）与 PTS 对齐。`✅已验证`
- 易错点：把 DTS 当显示时间；多数播放调度看 PTS。`✅已验证`

## 3.5 OpenGL 渲染管线认知
- 结论：可编程核心阶段是顶点/片元（在 ES 2.0/3.x 语境下）。`✅已验证`
- 易错点：把“DrawCall 数量”当唯一性能指标；状态切换、带宽、过度绘制同样关键。`✅已验证`

## 3.6 压缩纹理与 KTX2/BasisU
- 结论：KTX2 是容器；BasisU（ETC1S/UASTC）是可在运行时转码的中间表示。`✅已验证`
- 易错点：把“体积更小”误解为“加载一定更快”；转码 CPU 成本需评估。`⚠️有条件成立`

## 3.7 WebGPU / WebGL 关系
- 结论：WebGPU 非 WebGL 小改版，而是更接近现代显式 GPU API 的新模型。`✅已验证`
- 易错点：把两者性能关系绝对化。真实表现依赖驱动、后端、任务类型。`⚠️有条件成立`
