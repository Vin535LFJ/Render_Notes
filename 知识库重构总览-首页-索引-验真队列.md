## 0. 学习顺序总览（D）

建议按以下路径学习：

1. **基础概念层**：采样/像素格式/容器编码/时间戳/坐标系与矩阵
2. **数据流与时序层**：Demux → Decode → Render，音画同步
3. **API 与实现层**：FFmpeg、MediaCodec、OpenGL ES、Surface 系统
4. **优化与调试层**：DrawCall、纹理压缩、FBO、零拷贝、功耗/延迟
5. **工程架构层**：多路播放器、跨平台渲染抽象、RenderGraph
6. **项目实战层**：testKTX → GPU_Native-Runtime 演进

## 4. 待验证清单（C）

1. **“某方案提升 X% 性能/降低 Y% 功耗”**：缺统一测试方法、设备型号、样本量。`needs_source`
2. **“全链路零拷贝”**：需逐平台给出 buffer ownership 与 copy trace。`needs_source`
3. **“TextureView 一定比 SurfaceView 慢很多”**：绝对化，需设备矩阵测试。`uncertain`
4. **WebRTC 3A 参数细节与效果结论**：需对应 AEC/NS/AGC 实现源码或论文。`needs_source`
5. **KTX2/BasisU 在所有场景加载更快**：需分离“包体下载收益”与“转码 CPU 成本”。`uncertain`
6. **跨平台 API 行为一致性断言**：需各平台官方文档逐条对照。`needs_source`

## 5. 错误/绝对化表述改写示例（F）

1. 原表述：`TextureView 的渲染延迟显著高于 SurfaceView。`
   - 建议改写：`在多数 Android 设备上，TextureView 由于合成路径和纹理更新机制，通常会带来更高的渲染开销；但具体延迟差异依赖设备和系统版本。`
   - 证据：Android 官方 `TextureView` 与 `SurfaceView` 文档。

2. 原表述：`视频纹理化播放就是零拷贝。`
   - 建议改写：`视频纹理化播放可显著减少 CPU 侧拷贝，但是否“零拷贝”取决于解码输出格式、驱动和图形栈是否触发隐式转换。`
   - 证据：Android `SurfaceTexture` / `AHardwareBuffer` / EGLImage 相关资料。

3. 原表述：`DrawCall 多就一定卡。`
   - 建议改写：`DrawCall 增多通常会提高 CPU 提交与 GPU 状态切换开销，但性能瓶颈还受 shader 复杂度、带宽、过度绘制和分辨率影响。`
   - 证据：OpenGL/GPU 优化实践文档。

4. 原表述：`KTX2 一定更快。`
   - 建议改写：`KTX2 通常能降低分发体积并提升纹理资产管理效率；运行时性能取决于目标 GPU 格式匹配度与转码成本。`
   - 证据：Khronos KTX2 规范与 Basis Universal 文档。
