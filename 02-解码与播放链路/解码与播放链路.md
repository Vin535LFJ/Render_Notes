## 1. 音视频核心管线与职责边界 ✅已验证

### 1.1 核心结论

音视频处理的底层管线具备绝对不可逆性：**解封装（Demux） → 解码（Decode） → 渲染（Render）**。各模块在物理与逻辑上严格隔离，严禁职责混淆。

### 1.2 模块职责定义与工程边界

| **模块**            | **物理职责**                                                 | **技术边界（不负责）**                   | **核心工业级组件**                                 |
| ------------------- | ------------------------------------------------------------ | ---------------------------------------- | -------------------------------------------------- |
| **解封装（Demux）** | 解析容器格式（MP4/FLV/MKV），分离出压缩的音视频编码码流（H.264/H.265/AAC）。 | 不参与任何媒体数据的解压缩计算。         | `FFmpeg (libavformat)` `MediaExtractor`            |
| **解码（Decode）**  | 将输入的压缩码流，通过算力还原为原始像素矩阵（YUV/RGB）或原始音频采样点（PCM）。 | 不感知视频容器封装格式，不处理多轨同步。 | `MediaCodec`（硬解） `FFmpeg (libavcodec)`（软解） |
| **渲染（Render）**  | 将原始物理数据投射到屏幕物理像素点，或驱动声卡发出声波物理震动。 | 不具备任何解压与格式纠错能力。           | `SurfaceFlinger` / `OpenGL` `AudioTrack`           |

### 1.3 工业级标准工程组合方案

**架构原则：FFmpeg 全格式解封装 + MediaCodec 硬件解码优先 + FFmpeg 软件解码安全兜底**。

- `FFmpeg` 的 `libavformat` 充当全格式拆包工，负责对各类非标或常规流媒体进行高兼容解封装。
- 拆出的视频 NALU 优先送入系统底层的 `MediaCodec`，调用物理 SoC 上的专用 VPU 芯片进行硬解码。
- 当硬解实例耗尽、驱动崩溃或遭遇非标 Profile 触发 Fallback 状态机时，自动无缝切换至 `FFmpeg (libavcodec)` 启动 CPU 软解，确保业务高可用。

### 1.4 MediaCodec 定位

MediaCodec 是 Android **系统级底层硬编/硬解 API**，**仅负责编解码计算**，无任何解封装能力。必须搭配 `MediaExtractor` 或 `FFmpeg` 解封装后才能使用。 ✅已验证

### 1.5 FFmpeg 架构与能力边界

FFmpeg 是**全功能音视频处理框架**，模块化覆盖全流程： ✅已验证

| 子库        | 职责            |
| ----------- | --------------- |
| libavformat | 解封装 / 封装   |
| libavcodec  | 软解码 / 软编码 |
| libavfilter | 音视频滤镜      |
| libswscale  | 图像格式转换    |

### 1.6 标准工程组合

**工业级标准方案：FFmpeg 解封装 + MediaCodec 硬解优先 + FFmpeg 软解兜底** ✅已验证

1. FFmpeg 全格式解封装，分离压缩码流
2. 优先 MediaCodec 硬解
3. 硬解失败自动切换 FFmpeg 软解

### 1.7 易错点

- 把"播放器"与"解码器"混为一谈——播放器是架构容器，解码器只是其中一个环节
- 认为 MediaCodec 能做解封装——它仅处理编解码，解封装需额外组件



## 2. 软解与硬解深度对比 ✅已验证

### 2.1 本质原理

- **硬件解码（Hard Decoding）**：强依赖设备 SoC 内部固化的**专用视频处理单元（VPU/ASIC 芯片）**，通过硬件电路自驱完成硬解码，几乎不消耗 CPU 算力核心。
- **软件解码（Soft Decoding）**：纯粹通过编写运行在 **CPU** 上的数学算法（如 C 语言实现的 H.264 解压逻辑）来完成计算，极度依赖 CPU 的多核算力及向量指令集（NEON/AVX2）。

### 2.2 数据流向与内存拷贝成本

- **硬解路径**：`音视频压缩码流 → VPU 硬件解码器 → Gralloc 物理显存（零拷贝） → 渲染`。
  - *深度解密*：VPU 解码完成后的 YUV 像素直接填充到由系统图形层（Gralloc）管理的硬件缓冲区中，通过 `BufferQueue` 共享文件描述符（File Descriptor）进行进程间安全传递。整个流程数据始终在显存物理页内，CPU 无需搬移任何像素点。
- **软解路径**：`音视频压缩码流 → 系统主内存 → CPU 解码算力 → 系统主内存 → 逐帧总线拷贝 → GPU 显存 → 渲染`。
  - *深度解密*：CPU 软解出来的 YUV 原始像素存储在普通的系统主存（RAM）中。在画面渲染时，必须调用 `glTexImage2D` 或 `glTexSubImage2D` 将几兆字节的像素通过系统物理总线逐帧拷贝到 GPU 的显存中，引发高昂的总线带宽消耗与发热。

### 2.3 多维度对比表 ✅已验证

| 评估维度           | FFmpeg 软解                                          | Android MediaCodec                                      | iOS VideoToolbox                                        |
| ------------------ | ---------------------------------------------------- | ------------------------------------------------------- | ------------------------------------------------------- |
| **物理硬件路径**   | CPU 通用计算核心，依赖 NEON/AVX2 指令集              | SoC 集成的专用 VPU（ASIC 芯片）                         | Apple 自研 SoC 专有 VPU                                 |
| **内存寻址空间**   | 物理主存（System RAM），由 CPU 编址                  | GPU 侧图形缓冲区（Gralloc/硬件合成层）                  | CoreVideo 像素缓冲区（CVPixelBuffer/IOSurface）         |
| **显存传输成本**   | 极高（每帧需 `glTexImage2D` 逐帧拷贝，耗费总线带宽） | 零拷贝（通过 `BufferQueue` 共享文件描述符实现内存共享） | 零拷贝（Metal 纹理直接绑定 `CVPixelBuffer` 物理内存页） |
| **系统功耗**       | 极高（CPU 核心长期高负荷，易发热降频）               | 极低（专用 VPU 功耗通常小于 CPU 软解的 10%）            | 极低（硬件高度优化，热辐射极小）                        |
| **多路并发能力**   | 受限于 CPU 物理核心数、主频及 L3 缓存命中率          | 受限于硬件 VPU 实例的硬件并发解码通道上限               | 受限于设备 GPU 编解码引擎的并发限制                     |
| **非标码流鲁棒性** | 极强（软件算法纠错，可通过修改源码任意容错）         | 极差（对非标 GOP、不规则 Profile 极易黑屏/挂起）        | 较好（系统框架有部分容错，但非标码流亦有异常风险）      |

### 2.4 正确表述 ⚠️有条件成立

**"硬解一定更快"属于过度绝对化。** 实际表现需综合考量以下因素：

- **设备差异**：低端 SoC 的 VPU 实例有限，多路并发时硬解可能反而不如软解灵活
- **码率与分辨率**：低分辨率场景下，软解的 CPU 开销可接受，硬解的初始化开销可能抵消收益
- **驱动实现**：部分厂商的硬解驱动存在 Bug，可能导致性能回退
- **编码格式**：硬解对格式支持有限，不支持的格式只能走软解

**正确表述**：在硬件支持的前提下，硬解通常在功耗、延迟和帧率稳定性上优于软解；但"一定更快"需附加设备、码率、分辨率、驱动和实现路径等条件限定。



## 3. 视频渲染双链路深度解构：Surface 直出 vs OpenGL 渲染 ✅已验证

在 Android 图形系统中，硬解码完成后的像素流动存在两条截然不同的渲染管线。

### 3.1 链路 A：视频直接播放（Surface 直出）详细流程

该链路追求极致的性能与低功耗，通常用于常规视频播放、长视频平铺。

1. **I/O 读取**：播放器数据源模块（如 ExoPlayer DataSource）从闪存（Flash ROM）将二进制流读入系统主存（RAM）。
2. **解封装**：通过 `FFmpeg` 或 `MediaExtractor` 剥离出 H.264 / H.265 压缩码流包 (NALU)。
3. **输入队列**：应用层调用 `mediaCodec.dequeueInputBuffer()` 获取空闲缓冲区，并用 `queueInputBuffer()` 将 NALU 压入解码队列。
4. **芯片解码**：物理 SoC 上的专用 **VPU（视频处理单元）** 接管计算，将压缩帧还原为原始 YUV 像素，直接写入由 Gralloc 分配的硬件显存缓冲区。
5. **直投上屏（核心技术点）**：应用层调用 `mediaCodec.releaseOutputBuffer(index, render=true)`。因为初始化 MediaCodec 时，绑定的宿主是物理 `SurfaceView` 提供的 `Surface`。解码器底层会顺着 `BufferQueue` 传送带，通过进程间通信（IPC）将该硬件缓存描述符直接抛给系统级合成服务。
6. **窗口硬件合成**：系统的 `SurfaceFlinger` 服务接收到通知，通过 **Hardware Composer (HWC)** 在屏幕对应的位置上“挖一个洞”，利用硬件叠加层直接将该块视频显存投射上去。**该过程完全不经过 App 进程的 View 树二次图形混合，App 无法拦截、修改或读取任何像素点。**

### 3.2 链路 B：视频纹理化（OpenGL 渲染）详细流程

该链路专为特效、滤镜、美颜和剪辑而生，控制权完全收回到 App 进程的 OpenGL 线程中。

1. **前置解码**：数据的读取、解封装、送入 MediaCodec、VPU 解码与链路 A 完全一致。**其核心差异在于：MediaCodec 初始化时绑定的 Surface，是由中间件 SurfaceTexture 衍生出来的虚拟通道。**
2. **中转至虚拟显存**：调用 `mediaCodec.releaseOutputBuffer(idx, true)` 后，像素数据被投递给 `SurfaceTexture` 并锁存在 GPU 的共享显存中，同时触发应用层设立的 `onFrameAvailable()` 回调。
3. **外部纹理绑定（OES）**：应用层的 OpenGL 线程响应回调，调用 `surfaceTexture.updateTexImage()`，将最新的这帧 YUV 硬解显存流直接绑定为一个 **OES 外部纹理（`GL_TEXTURE_EXTERNAL_OES`）**。同时利用 `glGetTransformMatrix()` 抽离出纹理坐标变换矩阵，用以自动矫正视频原生附带的旋转角度。
4. **离屏格式转换（FBO 中转）**：由于 OES 外部纹理本身属于系统扩展纹理，在 OpenGL 体系中不支持跨纹理采样、无法直接回读像素、且内部仍保留 YUV 属性。OpenGL 线程此时会绑定一个 **FBO（帧缓冲区对象）** 执行一次离屏渲染。利用一段转换着色器（Shader）脚本，一气呵成将 OES 纹理画到一张标准的 **2D RGBA 纹理（`GL_TEXTURE_2D`）** 上。
5. **特效链级联加工（Effects Chain）**：标准的 2D 纹理成为完美的像素加工原料，在多个 FBO 特效节点之间流动：
   - **顶点着色器（Vertex Shader）**：通过修改顶点坐标矩阵，实现视频的分屏、镜像、动态裁剪、旋转与伸缩变换。
   - **片元着色器（Fragment Shader）**：通过逐像素采样算法，运行美颜（双边滤波/高斯模糊）、滤镜（读取 LUT 颜色查找表矩阵进行色彩强映射）、动效（灵魂出窍、幻影抖动）。
6. **UI 混合上屏或导出**：
   - *预览模式*：加工完毕的最终 2D 纹理通过 `eglSwapBuffers()` 绘制到 `TextureView` 上。它作为 App 控件树里的一个普通 View 节点，与周边的点赞按钮、UI 文字一起，由 UI 线程的 `RenderThread` 进行常规 GPU 混合，最后打包提交给 `SurfaceFlinger` 上屏。

### 3.3 两者的血缘联系与本质区别

| **对比维度**     | **链路 A：视频直接播放（Surface 直出）**                     | **链路 B：视频纹理化（OpenGL 渲染）**                        |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **物理芯片依赖** | 必须依赖底层 SoC 上的硬件 VPU 编解码单元。                   | 完全一致，同样依赖硬件 VPU 进行最初的图像解压。              |
| **内存拷贝开销** | **显存级零拷贝**。通过 `BufferQueue` 共享物理描述符，数据绝不回流 CPU。 | **显存级近零拷贝**。数据全程锁在 GPU 显存内，仅在 OES 转 2D 时引发显存内部像素格式转换。 |
| **画面控制权**   | **系统底层自治**。App 无法“偷看”和干预像素，无可编程特效入口。 | **App 进程绝对控制**。应用层可通过编写 Shader 着色器，进行像素级任意修改与特效加工。 |
| **宿主渲染载体** | `SurfaceView`（拥有独立的 Surface 窗口，与 App 的 View 树物理隔离）。 | `TextureView` 或自定义 `GLSurfaceView`（作为常规 View 节点参与层级混合）。 |
| **中间关键桥梁** | 物理 `Surface`                                               | `SurfaceTexture` + 虚拟外部 OES 纹理                         |
| **性能与功耗**   | **极低、极省电**。链路短，免去 GPU 纹理二次采样与 View 树图形重绘开销。 | **较高**。由于引入多级 FBO 中转与复杂的 Shader 像素密集型计算，GPU 电流消耗明显增大。 |
| **核心商用场景** | 传统播放器（爱优腾、奈飞电影播放）、本地高清万能播放器、电视直播大屏。 | 短视频特效（抖音/快手）、社交直播美颜挂件、视频剪辑合成器（剪映/秒剪）。 |

## 4. 工业级音视频工具链全景分工 ✅已验证

在工业级短视频剪辑与特效工程中，各类开源库与系统组件紧密协作，其严密的分工如下：

```
[ExoPlayer / IjkPlayer 播放器内核大管家] (控制状态机、音画同步、PTS 时间戳调度)
      │
      ├─► [FFmpeg libavformat] ────────────────► 执行解封装 (高效分离 MP4/FLV 容器)
      │
      ├─► [MediaCodec / libavcodec] ───────────► 执行解码 (VPU 芯片硬解 / CPU 多核软解)
      │
      ├─► [SoundTouch / Sonic 算法库] ─────────► 处理音频 (剪辑轨道变速不变调 PCM 处理)
      │
      └─► [EGL 系统总线环境配置] ────────────────► 搭建图形总线 (绑定 Android 窗口与 GPU 图形上下文)
            │
            └─► [OpenGL ES 3.0 管线] ──────────► 图形像素加工 (运行着色器、FBO 离屏级联特效链)
```

1. **ExoPlayer / IjkPlayer（播放器内核大管家）**：
   - *实际分工*：它们自身不具备解码能力和画图能力。它们是“工程胶水”，负责驱动解封装器，建立 Master Clock（主时钟），精准调度音视频 Track 的 PTS（显示时间戳）对齐，并向应用层暴露统一的 Play/Pause/Seek 状态机接口。
2. **FFmpeg（解封装绝对核心）**：
   - *实际分工*：其 `libavformat` 子库是工业界公认的拆包组件，负责将形态各异的媒体容器（MP4、MKV、FLV、M3U8）拆开，剥离出裸视频 NALU 序列输送给解码模块。
3. **SoundTouch / Sonic（音频时域压扩算法库）**：
   - *实际分工*：剪辑场景特有的“变声器”。当用户在编辑轨道上进行 0.5x 慢放或 2.0x 快放时，它负责在音频 PCM 原始数据输入 AudioTrack 播放前，执行时域重叠相加（WSOLA）等压扩算法，**确保改变视频播放速度的同时，维持音频的原声调不变尖（不变成小黄人音）、不沉闷**。
4. **EGL（嵌入式图形总线接口）**：
   - *实际分工*：OpenGL 本身是一套纯粹的数学渲染跨平台框架，并不知晓 Android 物理屏幕的存在。EGL 扮演了桥梁与总线角色，负责向 Android 底层系统申请图形上下文（Context）、配置本地显示连接（Display）及渲染表面（Surface），并保障多线程切换时显存数据的互斥与稳定。

| 挑战 | 核心痛点 | 详见 |
|------|----------|------|
| 1. 精准 Seek | 默认 Seek 只能跳到最近 I 帧，剪辑画面对不齐 | §8.2 → 1 |
| 2. 视频倒放 | GOP 帧间压缩依赖前后帧，无法直接倒序解码 | §8.2 → 2 |
| 3. 变速不变调 | 视频变速后音频不处理会变尖/变沉闷 | §8.2 → 3 |

### 5.1 精准 Seek（Frame-accurate Seek）

- **痛点**：默认的 Seek 只能跳转到距离目标时间点最近的关键帧（I帧），会导致剪辑时画面对不齐。
- **攻坚方案**：解封装器（如 FFmpeg）执行 `avformat_seek_file` 定位到目标 PTS **之前** 的最近一个 I 帧；随后解码器进入 **高速静默解码状态**，解出的 B/P 帧全部执行 `mediaCodec.releaseOutputBuffer(index, false)`（`render=false` 丢弃而不渲染）；直到解码帧的 PTS 正好对齐用户手指的目标 PTS 时，再恢复渲染管线，达成帧级精准度。

### 5.2 视频倒放（Video Reverse）

- **痛点**：视频文件由于 GOP 帧间压缩结构（P/B帧严格依赖前后帧），无法像音频一样直接倒序读取解码。
- **攻坚方案**：
  - *短视频/内存充裕方案*：建立 **Jitter Buffer 帧缓存队列**。正向解码一整个 GOP 组，将解码后的原始纹理临时存在显存/内存中，然后反向读取纹理序列进行 OpenGL 渲染。
  - *长视频/全画质方案*：导入阶段进行“预转码”，将原视频重编码为 **全 I 帧（Intra-only）** 视频流，使每一帧都能独立解码，从而支持无缝任意正倒放。

### 5.3 统一预览与导出（生成）双链路闭环

为了保证用户在剪辑时“所见即所得”，预览和导出两条管线必须共享同一套 OpenGL 特效 Shader 代码：

| **评估阶段**        | **预览链路（用户边剪边看）**              | **导出链路（生成最终 MP4 文件）**                           |
| ------------------- | ----------------------------------------- | ----------------------------------------------------------- |
| **1. 解封装内核**   | FFmpeg / MediaExtractor 拆出音视频压缩流  | 完全相同，保证底层数据源一致。                              |
| **2. 解码器机制**   | MediaCodec 硬件解码优先                   | 完全相同。                                                  |
| **3. 纹理中转**     | SurfaceTexture 生成 OES 外部纹理          | 完全相同。                                                  |
| **4. 特效渲染器**   | **运行预览版 Shader（美颜、滤镜、剪裁）** | **运行完全相同的 Shader（确保所见即所得）**。               |
| **5. 渲染终点站**   | 绑定并绘制到 **`TextureView`** 的 Surface | 绑定并绘制到 **`MediaCodec (编码模式)`** 的 Input Surface。 |
| **6. 后续收尾**     | 底层 `SurfaceFlinger` 硬件合成上屏显示    | 编码器压缩像素 ➔ **`MediaMuxer`** 封装为新 `.mp4` 文件。    |
| **7. 核心技术指标** | 追求高帧率稳定性、低延迟，允许部分丢帧    | 追求绝对不丢帧、每一帧精准编码、时间戳完美对齐。            |



## 6. Android 渲染组件体系

### 6.1 Surface 体系四组件 ✅已验证

| 组件               | 核心本质                       | 关键特性                             | 适用场景                      |
| ------------------ | ------------------------------ | ------------------------------------ | ----------------------------- |
| **Surface**        | 底层硬件绘图缓冲区             | 无 UI 层级，仅承载像素数据，不可见   | 解码/渲染底层数据载体         |
| **SurfaceView**    | 绑定 Surface 的独立渲染窗口    | 独立 UI 主线程，渲染优先级最高       | 低延迟纯播放、高帧率视频      |
| **TextureView**    | 封装 SurfaceTexture 的 UI 控件 | 与普通 UI 同层级，支持旋转/缩放/叠加 | 视频+UI 交互、轻量特效播放    |
| **SurfaceTexture** | Surface→GPU 纹理转换器         | 无可视化界面，生成 OES 外部纹理      | 纹理化播放、OpenGL 联动中间件 |

**组件关系（生产者-消费者模型）**：

Android 图形栈的 `BufferQueue` 采用生产者-消费者解耦：生产者把图像帧送入队列，消费者取走渲染，二者通过 Buffer 插槽（slot）轮转，天然支持异步、跨进程、跨线程。

| 角色 | 组件 | 职责 |
|------|------|------|
| 生产者 | `Surface`（`BufferProducer` 接口） | 解码器（MediaCodec）/绘制端向其提交已填充的 Buffer |
| 队列中枢 | `BufferQueue` | 持有若干 Buffer 插槽，协调生产者入队与消费者出队，管理 Buffer 的空闲/已入队/被持有三态流转 |
| 消费者 | `SurfaceTexture`（`BufferConsumer` 接口） | `updateTexImage()` 触发取帧，将 Buffer 内容绑定为 OES 外部纹理 |
| 上屏 | `SurfaceView` / `TextureView` | 物理渲染载体：SurfaceView 绑定独立物理 Surface（HWC Overlay 上屏）；TextureView 让消费侧的纹理参与 View 树 GPU 合成 |

> 关键理解：`Surface` 与 `SurfaceTexture` 是 **BufferQueue 一体两面**的生产者/消费者句柄——同一个 `BufferQueue`，一端被解码器当 `Surface` 写帧，另一端被 GL 当 `SurfaceTexture` 读帧。这是"零拷贝"得以实现的传输机制基础 ✅已验证

### 6.2 SurfaceView vs TextureView 底层对比 ✅已验证

| 评估指标     | SurfaceView                                                  | TextureView                                                  |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **合成机制** | 底层 `SurfaceFlinger` 作为独立的 Hardware Overlay（硬件覆盖层）进行多物理窗口的硬件合成 | App 的 UI Toolkit 在 View 树绘制阶段，将图像作为普通纹理与其他 UI 控件一起进行 GPU 合成 |
| **驱动线程** | 独立的后台非 UI 渲染线程（不抢占 UI 刷新主时间片）           | 主 UI 线程的 RenderThread，易受其他 UI 复杂布局重绘和动画执行影响 |
| **物理延迟** | 极低（直接写入物理窗口双缓冲，近乎零开销）                   | 较高（通常存在 1~3 帧的 View 树二次合成及管线同步延迟）      |
| **交互能力** | 极差（无法完美支持三维倾斜、局部遮罩裁切、半透明淡入淡出及滚动视图混排） | 极强（支持各种矩阵变换、全范围动画、任意网格剪裁及 Alpha 物理层过渡） |
| **线程安全** | 极高（与 UI 主管线解耦，物理隔离，不发生死锁）               | 偏低（高频次 Buffer 交互极易引发多线程同步死锁与 EGLSwap 悬挂） |
| **功耗表现** | 极佳（Hardware Overlay 合成无需 GPU 重画像素，能效比极高）   | 一般（频繁的纹理采样与二次图形混合增加 GPU 电流消耗）        |

**选择原则**：

- 追求极致低延迟、低功耗的纯播放场景 → SurfaceView
- 需要 UI 变换、动画叠加、滚动容器嵌入 → TextureView

### 6.3 直出与纹理化播放 ✅已验证

| 维度             | 直出播放（Surface 直出）     | 纹理化播放（OpenGL 渲染）                |
| ---------------- | ---------------------------- | ---------------------------------------- |
| **数据存储位置** | Surface 硬件帧缓冲区（显存） | GPU 显存纹理单元                         |
| **可编程性**     | 无可编程入口，无法加特效     | 可编程，支持像素级修改                   |
| **底层本质**     | 硬件直出，无中间处理         | GPU 可编程渲染，支持滤镜/美颜/合成       |
| **链路特征**     | 链路短，通常低开销           | 链路长，增加 GPU 纹理转换+着色器运算开销 |

**关于延迟与拷贝的严谨表述** ⚠️有条件成立：

- 任何"固定延迟区间""全程零拷贝"都必须加设备与系统版本条件
- SurfaceView 直出延迟通常低于 TextureView，但具体数值受设备合成策略、缓冲区数量、系统版本影响
- "TextureView 一定慢很多"属于绝对化错误表述，实际差异取决于具体使用场景和设备







## 7. 视频纹理化与 OES 外部纹理

### 7.1 OES 纹理原理 ✅已验证

**完整流程**：

1. 解封装 → MediaCodec 硬解
2. 绑定 **SurfaceTexture** 替代 Surface
3. 硬件解码数据写入 SurfaceTexture
4. 自动生成 **OES 外部纹理**（GPU 显存）
5. OpenGL 读取纹理渲染到屏幕

**核心原理**：SurfaceTexture 将硬件解码 YUV 数据转为 `OES_EGL_image_external`（`GL_TEXTURE_EXTERNAL_OES`）纹理，直接供 OpenGL 使用，**无 CPU 转码**。

### 7.2 与 GL_TEXTURE_2D 的关键差异

- `GL_TEXTURE_EXTERNAL_OES`：专门用于接收视频解码流，内部数据可能是 YUV 或 RGB 变种，采样参数受限，无法直接进行复杂的跨纹理重复采样或直接回读。
- `GL_TEXTURE_2D`：标准二维纹理（通常为 RGBA 格式），支持完整的采样参数和各种 OpenGL 混合操作。
- **两者的桥梁**：不能直接互换，必须通过离屏渲染（FBO）技术，在 Shader 中将 OES 纹理渲染到标准的 `GL_TEXTURE_2D` 纹理上，才能进入后续的特效链。

### 7.3 零拷贝的真实含义 ⚠️有条件成立

**Surface 直出**：全程零拷贝，数据直接映射显存。VPU 将解码像素填充到 GPU 驱动直接管理的 Gralloc 缓存中，通过 `BufferQueue` 机制统一管理，渲染引擎通过 OES 外部纹理扩展直接读取该显存区域，无需 CPU 参与主存到显存的数据物理搬移。

**纹理化播放**：显存内一次格式转换，接近零拷贝。SurfaceTexture 将解码帧转为 OES 纹理时，数据仍在 GPU 显存内流转，但可能存在显存内 YUV 到 RGB 的格式转换操作。

### 7.4 工业级特效/滤镜核心管线（OES ➔ FBO ➔ 2D纹理） ✅已验证

在处理多重特效、美颜及滤镜叠加时，直接操作 OES 纹理会受到极大限制。行业标准的架构会引入 **FBO（Framebuffer Object）离屏渲染** 建立特效链：

```
[MediaCodec 硬解]
       │
       ▼
[SurfaceTexture] ➔ 转换为 [OES 外部纹理 (GL_TEXTURE_EXTERNAL_OES)]
       │
       ▼ (第一道转换 Shader：处理旋转矩阵与 YUV➔RGB 转换)
[FBO 离屏渲染缓冲区]
       │
       ▼ (输出为标准的普通的 [GL_TEXTURE_2D] 纹理)
┌──────────────────────────────────────────────┐
│  进入特效链 (Effects Chain)                   │
│  ➔ 顶点着色器 (Vertex Shader)：裁剪、分屏、伸缩变换│
│  ➔ 片元着色器 (Fragment Shader)：美颜、LUT调色、动效│
└──────────────────────────────────────────────┘
       │
       ▼ (加工完成的标准 2D 纹理)
┌──────┴───────────────────────────────────────┐
▼ (预览状态链路)                                 ▼ (导出/生成状态链路)
渲染到 TextureView/GLSurfaceView 上屏播放         渲染到编码器配置的 Input Surface 执行硬编
```

### 8.2 视频剪辑场景下的三大核心挑战与解决策略 ✅已验证

视频剪辑类项目（如剪映、视频编辑器）不仅要求播放，还对帧级别的控制提出了极高的要求，必须在架构层攻克以下技术点：

#### 1. 精准 Seek（Frame-accurate Seek）

- **痛点**：默认的 Seek 只能跳转到距离目标时间点最近的关键帧（I帧），会导致剪辑时画面对不齐。
- **方案**：解封装器（如 FFmpeg）执行 `avformat_seek_file` 定位到目标 PTS **之前** 的最近 I 帧；随后解码器进入 **高速静默解码状态**，解出的 B/P 帧全部执行 `mediaCodec.releaseOutputBuffer(index, false)` 丢弃而不渲染；直到解码帧的 PTS 正好对齐目标 PTS 时，恢复渲染管线，达成帧级精准度。

#### 2. 视频倒放（Video Reverse）

- **痛点**：视频文件由于 GOP 帧间压缩结构（P/B帧依赖前后帧），无法像音频一样直接倒序读取解码。
- **方案**：
  - *短视频/内存充裕方案*：建立 **Jitter Buffer 帧缓存队列**。正向解码一整个 GOP 组，将解码后的原始纹理临时存在显存/内存中，然后反向读取纹理序列进行 OpenGL 渲染。
  - *长视频/全画质方案*：导入阶段进行“预转码”，将原视频重编码为 **全 I 帧（Intra-only）** 视频流，使每一帧都能独立解码，从而支持无缝任意正倒放。

#### 3. 音频变速不变调（Time Stretching）

- **痛点**：剪辑时视频进行 0.5x 慢放或 2.0x 快放，音频如果不做特殊处理，会导致声调变尖（小黄人音）或变沉闷。
- **方案**：引入 **SoundTouch** 或 **Sonic** 开源音频处理库，在音频解码出 PCM 原始数据后、进入 AudioTrack 播放前，利用时域压扩算法（如 WSOLA 算法）对 PCM 采样点进行重组，确保**改变播放速度的同时维持原声调**。

### 8.3 预览与导出（生成）双链路闭环架构对比表 ✅已验证

为了保证用户在剪辑时“所见即所得”，预览和导出两条管线必须共享同一套 OpenGL 特效 Shader 代码：

| **3. 纹理中转**     | SurfaceTexture 生成 OES 外部纹理          | 完全相同                                                  |
| **4. 特效渲染器**   | **运行预览版 Shader（美颜、滤镜、剪裁）** | **运行完全相同的 Shader（确保所见即所得）**               |

### 8.4. 高可用：硬解向软解自动回退状态机 ✅已验证

当视频流分辨率突变、网络丢包导致 GOP 严重不完整，或底层 SoC 硬解实例耗尽时，硬件解码器会崩溃或抛出不可恢复的硬件错误。播放器设计了如下无感知 Fallback 状态机机制：

1. **拦截硬解错误**：监控层精准捕获 MediaCodec 抛出的底层物理硬解异常（如 `CodecException`）。
2. **画面挂起拦截**：即刻锁存当前发生崩溃的黄金进度时间戳 `PTS_fail`，控制视频渲染器强行切入“冻结最后一帧（Freeze）”状态，物理隔绝黑屏或白闪现象。
3. **音频静音渐变**：对当前正在播放的音频执行包络平滑的静音渐变（Fade Out），从物理层阻断因硬解切换造成的瞬间爆音或瞬态杂音。
4. **销毁硬解上下文**：优雅安全地完全释放掉崩溃的硬件解码器实例及其关联的虚拟 Surface 资源。
5. **初始化软解器**：以毫秒级响应速度快速初始化 `libavcodec` 软件解码器上下文。
6. **Seek 至安全点**：命令解封装器快速定位到 `PTS_fail` 绝对时间轴之前的最近一个安全 I 帧（关键帧）。
7. **静默追赶**：软件解码器全负荷运转，开启“静默解码（不渲染输出）”模式高速消耗并丢弃陈旧帧，直至解码 PTS 严格追赶平齐至 `PTS_fail`。
8. **恢复渲染**：重新激活视频渲染管线，音频轨道同步执行渐入淡化（Fade In），完成终端用户无感知的高可用过渡。

## 附录：核心音视频术语速查

- **FBO (Framebuffer Object)**：帧缓冲区对象。OpenGL 中的离屏渲染组件，作为中转托盘，是建立多重特效级联链（Effects Chain）的核心。
- **OES 纹理 (GL_TEXTURE_EXTERNAL_OES)**：Android 特有的外部图形扩展纹理，直接映射硬解芯片的 YUV 显存空间，是直出数据进入 OpenGL 世界的唯一入口。
- **LUT (Look-Up Table)**：颜色查找表。一幅 2D/3D 的颜色映射图，Shader 通过读取 LUT 像素点将原视频色彩进行像素级替换，是工业级高级滤镜的核心技术。
- **BufferQueue**：Android 系统的核心图形缓冲队列，连接图像生产者（如 MediaCodec）与图像消费者（如 SurfaceTexture / SurfaceFlinger）的纽带。
- **Demux**：解封装，指从 MP4/FLV 等文件容器中彻底分离音视频码流的逆向动作。
- **PTS (Presentation Time Stamp)**：显示时间戳，用于严格指导每一帧像素在何时投射到屏幕上。
